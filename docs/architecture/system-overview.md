# Обзор системы

Liber Novus строится как многослойная система, где генеративный AI является только одним из этапов, а не центром архитектуры.

## Высокоуровневая схема

```text
Telegram / Mini App / будущие клиенты
            ↓
      delivery layer
            ↓
      backend API
            ↓
      service layer
            ↓
  deterministic processing
            ↓
  analysis orchestrator
            ↓
  provider abstraction
            ↓
       PostgreSQL
```

## Основные слои

- Delivery layer принимает пользовательский ввод и возвращает результат.
- Backend API задает контракт внешнего доступа.
- Service layer владеет бизнес-логикой.
- Deterministic layers подготавливают, агрегируют и валидируют данные.
- Analysis orchestrator координирует многостадийное рассуждение.
- Provider abstraction изолирует конкретные LLM-провайдеры.
- PostgreSQL является единственным source of truth.

## Архитектурный принцип

Все, что может быть детерминированным, должно оставаться детерминированным: lifecycle сессий, summary, prompt compilation, validation, извлечение повторяемых слов и базовая агрегация.

LLM получает только структурированный, ограниченный и проверенный контекст. Если вывод невозможно объяснить через исходные данные, prompt, стадию reasoning и provider response, архитектура считается сломанной.

## Независимость от клиентов

Telegram является текущим транспортом, но бизнес-логика не должна зависеть от Telegram. Система должна поддерживать будущий web-клиент, Mini App и другие интерфейсы без переписывания core backend.
